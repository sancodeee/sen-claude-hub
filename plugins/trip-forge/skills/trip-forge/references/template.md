## 二、核心体验诉求（决定取舍方向，优先级最高）

> **整个行程中，至少安排一段完整的「慢下来」时光**（约 60–90 分钟）——海边 / 山顶 / 湖边 / 咖啡馆，停下来发呆、散步、看景。请挑全程中最适合的那一天、最适合的那个时段来安排（比如景色最好的一天、或天气最佳的傍晚），把它做成一个**亮点**，而不是每天机械凑时长。我看重的是**现场氛围、安静度、舒服程度和真实自然感**，而不是出片率。**此诉求优先级高于「多去一个景点」。当它与塞满点位冲突时，砍点位，保这段慢时光。**

其余每天保持整体松弛、不赶路即可，不必再硬性留固定时长。

## 三、个人偏好（说明"我要什么"）

**1. 行程节奏**
- 不要特种兵、不要早出晚归；
- 每天最多 2–3 个核心点位；
- 主动预留吃饭、休息、喝咖啡、临时调整的时间；
- 宁愿少去，也要体验舒服。

**2. 景点取向**
- 优先自然风景、本地生活感、体验真实的地方；
- 避开网红排队点、过度商业化景区、纯打卡机位；
- 不推荐仿古街、义乌小商品式夜市、低质商业街；
- 若推荐热门景点，必须说明**最佳时段**和**避开人流的具体方法**。

**3. 看景偏好（承接第二部分）**
- 优先有氛围、浪漫惬意、人少安静的原生态海岸 / 礁石 / 森林 / 湖边 / 草地；
- 适合日出 / 日落 / 夜景的，请**主动判断是否值得安排**，并回答：最佳时间、推荐点位、是否需早起绕路、人多不多、交通是否方便、天气差时的替代方案；
- **不要**为看景安排高强度爬山、长时间暴晒、危险野路、过度偏僻处；若风景好但代价大，请提前说明取舍。

**4. 餐饮偏好**
- 拒绝排队很久的网红店；
- 优先本地人常吃的餐馆、居民区小店、市场、家常菜；
- 不要只堆店名，要说明**"吃什么类型 + 去哪个区域吃"**；
- 推荐具体店时，说明：为什么值得去、点什么、是否易排队、有无平替；
- 避开景区门口店、游客价店、套路店。

**5. 交通偏好**
- 优先省心方案；短距离能打车就别强行公交；
- **每段路都要写明建议方式 + 预计耗时**；
- 公交绕路就直接建议打车；
- 跨城交通请对比高铁 / 飞机 / 自驾 / 租车，并给出**最推荐选择**。

**6. 住宿偏好**
- 推荐交通方便、吃饭方便、晚上能散步的**具体片区**（不要只说"住市中心"）；
- 说明为什么这个片区适合本次同行人；
- 给出选酒店原则（离车站 / 地铁 / 海边 / 老城区的距离建议）；
- 明确**不推荐**哪些片区及原因。

**7. 体验偏好**
- 要有当地生活感，不只逛景区；
- 可安排散步、咖啡馆、市场、老街、日落点、轻松机位；
- 不安排连续暴走、长时间暴晒；
- 适合拍照处，请说明最佳时间、机位方向、人多不多。

## 四、强制输出约束（规定"怎么交付"）

- **【强制】** 内容**结构化**（表格 / 分点 / 卡片），不要写成散文；
- **【强制】** 不要泛泛而谈，每条都要能直接照着做；
- **【强制】** 每个景点都要回答**"为什么去 / 怎么去 / 玩多久 / 怎么避坑"**，不要只给清单；
- **【强制】** 所有交通段必须写清方式（打车 / 公交 / 地铁 / 步行 / 自驾）+ 耗时；
- **【强制】** 结论必须明确，不要模棱两可；
- **【强制 · 信息置信度分级】** 凡涉及价格、营业时间、预约规则、班次、季节性开放等易变信息，必须逐条标注置信度：
  - ✅ 较稳定（如常识性地理、长期固定景观）
  - ⚠️ 需出行前确认（如门票价、营业时间、预约名额）
  - ❓ 不确定，强烈建议查证。**不要把不确定信息当作既定事实输出。**
- **【强制 · 信息时效】** 门票、营业时间、交通班次、季节性开放、天气、预约政策等**必须基于最新联网核实**而非凭记忆；引用关键数据注明来源与日期，无法核实标 ❓ 并提醒二次确认。

## 四+、详细度下限（每个条目都写到「拿来即用」，最重要）

> ⚠️ **这是本攻略最容易翻车的地方。** 报告的价值取决于「用户拿着它还要不要再查资料」。停在「类型 + 区域」的概括层（如「本地口碑鲅鱼水饺老字号」「打车去养马岛」）等于没写完。**「松弛」是行程节奏，不是信息密度**——绝不能因为风格松弛、或因为某条没逐一核实，就把信息量也砍掉。能查到的长期稳定信息（店名、价格区间、车次时刻、公交线路号、门票数字、所在街区）**必须写出来并标置信度**，不许用「价格浮动 / 以当日为准」当借口省略具体数字。

**每类条目的最小颗粒度（达不到就算没写完）：**

- **跨城交通**：给**具体车次 / 航班号 + 发车到达时刻 + 时长**（里程可选）；列去程与返程各至少一个具体班次；二等座 / 票价给区间并标 ⚠️；再对比高铁 / 飞机 / 自驾 / 租车并给最推荐。
- **当地交通**：每段写**方式 + 耗时 + 大致费用区间**；点名值得一坐的观光线路（如海景公交线路号）与岛上 / 景区内的具体代步方式及价格（如电动车 X 元/小时）。
- **景点**：**具体名称 + 门票数字 / 开放时间 + 所在片区 + 到达方式与耗时 + 建议时长 + 最佳时段**；不要只给大类。
- **餐饮**：给**1–2 家具体店名 + 招牌菜 + 人均区间 + 所在街区 + 是否易排队 + 平替**；只说「找本地老店」不算数。
- **住宿**：**具体片区名 + 选房原则（距海 / 站 / 老城的步行分钟或公里）+ 点名不推荐的片区及原因**。

**反面 → 正面对照（照正面写）：**

| ❌ 太概括（不合格） | ✅ 拿来即用（合格） |
| --- | --- |
| 打车去养马岛，岛上可租车 | 市区打车 30–40 分钟到岛口；岛已限行需换乘，岛外租电动车环岛约 30 元/小时（带棚约 40），共享单车 10 元/30 分钟 ⚠️ |
| 本地口碑鲅鱼水饺老字号，人均约 55 | 三合园饺子馆（二马路），鲅鱼水饺招牌、皮薄大馅，人均约 50–60，周末排队，平替同片区任一本地水饺馆 ⚠️ |
| 高铁直达，约 6 小时 | 去程 G1087 北京南 11:22→烟台 16:33（5h11m，929km，途经蓬莱）；返程 G1084 烟台→北京南，二等座约 300+ 元/人，周末早订 ⚠️ |

**点位覆盖（先穷尽再取舍）：** 先尽量列全候选点位与「可选 / 替代 / 加分项」（如日出点、室内博物馆、观光公交、备选海岸），再按本次风格做取舍；**砍掉的点要写明为什么砍**（如「长岛需渡轮太赶，用养马岛替代」），而不是直接不提——让用户看到你权衡过，而非信息缺失。

## 五、六个维度（同等完整，不得精简任何一维）

> ⚠️ **完整度优先：** 六个维度同等完整。若单轮篇幅不足，按分段写入文件解决——**用分段落盘解决篇幅，绝不用压缩内容解决**。维度 4–6（吃喝 / B计划 / 物资）与 1–3 一样必须详尽可执行。每个条目的颗粒度套用上方「四+、详细度下限」。

### 维度 1 · 极简行程表

按「上午 / 午餐 / 下午 / 晚餐 / 晚上」时间轴排，每天不排满，保留休息与机动时间。可去可不去的标「可选」。

> **【强制 · 三餐入表】** 每天行程表**必须含一行午餐、一行晚餐**（作为独立时间行落进表里，不要只把吃饭甩到维度 4）。餐行套用「四+、详细度下限」的餐饮颗粒度：**具体店名/类型 + 招牌菜 + 人均区间 + 所在街区**（可注「详见吃喝指南」与维度 4 呼应）。
> **例外要显式**：某餐确在途中解决（抵达/返程在高铁、飞机上），仍保留该餐行并写明「高铁上解决 / 12306 订餐 / 自带」，让省略**有意且可见**，而非缺失。

行程表字段（概览，不塞景点细节）：天数 | 时间段 | 安排 | 天气依赖度 | 建议交通 | 预计耗时 | 游玩时长 | 人流时段提示 | 雨天替代。餐行的「建议交通」写从上一点位过去的走法+耗时，「雨天替代」写室内不受影响或同片区平替店。

> **「天气依赖度」每个点位必填**，三档：
> - ☀️ **强**：靠晴天 / 好光线（看海、日落、山顶远眺、夜景），阴雨大打折扣
> - ⛅ **中**：阴天也能去但效果减分
> - 🏠 **弱**：室内或不挑天气（博物馆、咖啡馆、市场、老街、室内美食）
>
> 「人流时段提示」「雨天替代」同为必填——对应避人流、看天气的需求，落进表格而非散落表外。

> 行程表只放概览，景点细节交由下方「景点详情卡」承接，确保手机不挤、信息不被简化。

#### 维度 1 · 景点详情卡（承接每个核心景点「四问」，不得省略）

> **【强制】** 每个**核心点位**单独一张详情卡（「可选」点位酌情），逐张写全：
> - **为什么去**：亮点 + 为什么适合本次同行人（承接核心诉求与偏好）；
> - **怎么去**：从住宿片区 / 上一点位出发的方式（打车 / 公交 / 地铁 / 步行 / 自驾）+ 预计耗时；
> - **玩多久**：建议游玩时长 + 最佳时段（含日出 / 日落 / 软光判断）；
> - **怎么避坑**：避人流具体方法、是否易排队、有无平替、风景好但代价大的取舍提醒；
> - **置信度与来源**：涉门票 / 营业时间 / 预约 / 班次等易变信息时就近标 ✅ / ⚠️ / ❓ 并注来源与核验日期。

#### 维度 1+ · 看天调度（把天气前置到排程里）

> **【强制】** 不要把固定行程甩给用户自己看天调整——结合出行日期天气预报（标 ⚠️ 需出行前再确认）主动给弹性排程：
> 1. **好天放刀刃上**：把「☀️ 强依赖」点位优先排在预报最晴朗的那天 / 时段集中使用；
> 2. **坏天换室内**：某天阴 / 雨则主推「🏠 弱依赖」（室内馆、美食、咖啡、市场、老街）；
> 3. **给对调开关**：明确告诉用户——现场与预报不符时，哪两天可对调及判断标准。
>
> ⚠️ **阴天 ≠ 晴天**：灰蒙蒙下看海、日落、远眺明显打折，遇阴天显式建议换不挑光线的安排，好景留晴天。

### 维度 2 · 交通住宿最优解

**直接给结论**，不模棱两可：
- 从【出发城市】到【目的地】最推荐方式，及高铁 / 飞机 / 自驾 / 租车各自适合场景；
- 当地更适合打车 / 公交 / 地铁 / 自驾 / 租车；
- 本次同行人怎么走**最省心 + 性价比最高**；
- 推荐住哪个**具体片区**、为什么方便；不推荐哪些片区及原因；选酒店原则。

### 维度 3 · 避坑与防雷清单

每条按格式，「直接照做」**必须是具体动作**（几点去 / 订哪家 / 走哪条路），不能是「建议提前规划」这类废话：

```text
不建议：____
原因：____
替代方案：____
直接照做：____（必须具体到动作）
```

覆盖：不必专程去的商业街 / 易踩坑景点 / 易被宰餐饮类型 / 不划算交通 / 易堵车人多时段。

### 维度 4 · 本地人吃喝指南

按分类输出，重点说"吃什么类型 + 去哪个区域"，并给出**如何判断一家店是否靠谱**：
早餐 / 午餐 / 晚餐 / 咖啡甜品 / 夜宵 / 当地必吃类型 / 不建议吃的类型（及原因）。

推荐具体店时注明：是否易排队、适合点什么、有无平替。

### 维度 5 · 突发情况 B 计划

逐项给**可直接执行的替代路线**（不要只说"改去商场"）：
1. 下大雨　2. 太热 / 太冷 / 风太大　3. 核心景点临时关闭　4. 人太多排队　5. 同行人累了不想走　6. 临时想看日出 / 日落 / 夜景 / 找地方安静看景喝咖啡　7. 餐厅排队太久　8. 交通延误 / 行程被打乱
